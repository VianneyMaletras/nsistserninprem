ESP32 Chess Clock: Building a Digital Timer with Dual OLED Displays

ARNAV VERMA
3 min read
·
Apr 12, 2025

Introduction

Chess clocks are essential tools for timed chess matches, ensuring fair play by allocating each player a fixed amount of time. In this tutorial, I’ll walk through creating a digital chess clock using an ESP32 microcontroller and dual OLED displays. This project features increment timing, customizable settings, and an intuitive interface.

Link to GitHub

Project Overview

Our chess clock uses:

  • ESP32 microcontroller
  • Two SSD1306 OLED displays (128x64 pixels)
  • Five push buttons for control
  • A clean, modular C++ codebase (GitHub Link)

Features

  • Dual displays: Each player has their own dedicated timer display
  • Time configuration: Set hours, minutes, and seconds for each player
  • Increment timing: Add time after each move (configurable)
  • Player controls: Dedicated buttons for each player to stop their timer and start their opponent’s
  • Visual feedback: Highlights time fields during configuration

Hardware Requirements

  • ESP32 development board
  • 2× SSD1306 OLED displays (with different I2C addresses)
  • 5× momentary push buttons
  • 5x 4.7K ohm resistors
  • Breadboard and jumper wires
  • USB cable for programming

Schematic

Software Architecture

I’ve organized the code into several modular files to improve readability and maintainability:

  1. ChessClock.h: Main header file with constants, enums, and function declarations
  2. ChessClockGlobals.h: Declarations of global variables used across files
  3. ChessClock.cpp: Core display functions and interrupt handlers
  4. ChessClockTimer.cpp: Timer functionality and settings configuration
  5. main.ino: Setup and main loop

Usage Instructions

  1. Power on: Connect the ESP32 to power
  2. Setup mode: The system starts in configuration mode
  3. Set player 1 time: Use Mode button to cycle through hours, minutes, seconds and use the increment or decrement button to increase or decrease the value.
  4. Set player 1 increment: Continue cycling to set increment values
  5. Set player 2 time and increment: Continue with Mode button
  6. Start game: After setup, both timers display their initial countdown times. The game begins when the player with the black pieces presses the button to start the white player’s timer.
  7. During play:
  • Player 1 makes a move, then presses their button
  • Player 1’s clock stops, and their increment time is added
  • Player 2’s clock starts running
  • Player 2 makes a move, presses their button
  • The cycle continues until one player runs out of time

Video Demonstration

Conclusion

This ESP32-based chess clock demonstrates how to create a practical timing device with a clean, modular codebase. The dual OLED displays and configurability make it suitable for serious chess play, while the modular design allows for easy expansion and customization.

The project showcases several important embedded programming concepts:

  • Interrupt handling for responsive user input
  • I2C communication with multiple devices
  • State management for different modes of operation
  • Modular code organization for maintainability

By studying this implementation, you can learn valuable techniques applicable to many other embedded projects requiring user interfaces and precise timing.

Challenges I Faced During the Build

  1. Handling Hardware Interrupts Reliably
    Working with external interrupts for player buttons required careful attention to ensure debounce wasn’t causing multiple triggers or erratic timer behavior.
  2. Managing Simultaneous Timers
    Implementing two independent timers that alternate accurately based on player input without overlap or delay was tricky, especially when switching states via interrupts.
  3. OLED Display Updates Without Flickering
    Getting both SSD1306 OLED displays to update smoothly without flickering or blocking other processes was a challenge. Optimizing display1.display() calls and managing screen refresh rates was important.
  4. Precise Button Debouncing
    Although using interrupts provided fast response, mechanical bounce in push buttons occasionally caused unwanted behavior, requiring software debounce logic or hardware solutions like capacitors.
